## 摘要
混合专家（MoE）模型已成为扩展大型语言模型的事实标准。为保持计算效率，现代MoE服务系统通常采用混合并行策略，将注意力阶段的数据并行（DP）与MoE阶段的专家并行（EP）相结合。然而，这种设计需要在注意力DP组与专家之间频繁进行全局同步屏障。在线服务中，请求到达率和序列长度的显著变化固有地导致DP不均衡，引发严重的同步停滞，从而降低首令牌时间（TTFT）和系统吞吐量。我们提出了ASAP，一个专门设计用于加速MoE模型预填充阶段的异步推理系统。ASAP解耦了注意力阶段和MoE阶段，并实现了完全异步的执行流水线。这是通过一套专门的异步通信原语以及跨请求调度和模型执行的四个协调优化实现的，这些共同消除了全局同步屏障。我们在CloudMatrix384超级节点上实现并评估了ASAP，结果表明，与最先进的同步服务解决方案相比，它将符合SLO的预填充吞吐量提高了90%。

## Abstract
Mixture-of-Experts (MoE) models have become the de facto standard for scaling large language models. To maintain computational efficiency, modern MoE serving systems typically employ a hybrid parallelism strategy, combining Data Parallelism (DP) for attention stages with Expert Parallelism (EP) for MoE stages. However, this design necessitates frequent global synchronization barriers between attention DP groups and experts. In online serving, significant variance in request arrival rates and sequence lengths inherently leads to DP imbalance, causing severe synchronization stalls that degrade Time-to-First-Token (TTFT) and system throughput. We present ASAP, an asynchronous inference system specifically designed to accelerate the prefill phase of MoE models. ASAP disaggregates the attention and MoE stages and implements a fully asynchronous execution pipeline. This is achieved through a suite of specialized asynchronous communication primitives and four coordinated optimizations across request scheduling and model execution, which collectively dismantle global synchronization barriers. We implement and evaluate ASAP on CloudMatrix384 super-nodes, demonstrating that it improves SLO-compliant prefill throughput by 90% compared to state-of-the-art synchronous serving solutions.

## 1. 核心问题与整体含义（研究动机和背景）
- **问题背景**：混合专家（MoE）模型（如DeepSeek-V3）已成为大模型主流架构。推理服务通常采用混合并行策略：注意力阶段使用数据并行（DP），MoE阶段使用专家并行（EP）。这要求注意力DP组与MoE设备之间频繁进行全局同步屏障。
- **核心问题**：在线服务中，请求到达率随机、序列长度分布极不均匀（生产日志显示从31到10万tokens，平均约5k），导致注意力DP组之间工作负载不均衡（DP Imbalance）。注意力计算复杂度为 O(∑s_i²) ，即使总token数相同，不同DP组的执行时间差异也很大。同步屏障迫使所有DP组等待最慢的组，造成严重的资源空闲和TTFT（首令牌时间）增加。
- **研究意义**：现有的同步推理系统无法消除这种由负载不均衡导致的执行气泡。本文首次提出完全异步的MoE预填充推理系统ASAP，旨在拆除全局同步屏障，让请求以各自的速度流动，从而同时降低延迟和提高吞吐。

## 2. 方法论
### 2.1 核心思想
- 物理上解耦注意力阶段和MoE阶段，将它们部署到不同的设备上。注意力设备数量 = D × T，MoE设备数量 = E。
- 设计异步通信原语，基于共享内存缓冲区实现非阻塞的1-to-N和N-to-1数据传输。
- 通过四个协调优化保持硬件利用率：长度感知批处理、双批交错、通信-计算重叠、气泡免除核调度。

### 2.2 关键技术细节
#### 异步通信原语
- **分布式共享内存抽象**：每个设备静态分配全局可见的共享缓冲区。发送方写入数据后立即返回，无需等待接收方确认；接收方独立轮询并取走数据。
- **四种原语**：
  - `async-dispatch-send`：注意力设备向多个MoE设备发送token（专家ID和隐藏状态）。
  - `async-dispatch-recv`：MoE设备轮询收到数据后取走并清除标志。
  - `async-combine-send`：MoE设备将专家结果写回注意力设备的共享缓冲区。
  - `async-combine-recv`：注意力设备轮询收集所有专家结果。
- **缓冲区结构**：详见论文Table 2。注意力设备缓冲区存放专家ID、结果和E位位图；MoE设备缓冲区存放token元数据、隐藏状态和DT位位图。

#### 长度感知批处理
- MoE阶段需要至少2k tokens（拐点）才能从内存受限转为计算受限的线性区。调度器聚合请求形成批大小超过该阈值的批次，同时无需考虑跨DP组负载均衡（因为异步执行已消除同步）。

#### 双批交错
- 为消除注意力设备在MoE处理期间的空闲，将两个批次（dual-batch）配对调度到一个注意力DP组。注意力设备交替处理两个批次的各层：当一个批次的注意力结果发送到MoE后，立即开始另一个批次的同一层计算。

#### 通信-计算重叠（三流设计）
- 在MoE设备上使用三条独立硬件流：一条主计算流执行MoE前向，两条通信流分别处理异步接收和异步发送。核心资源分区：所有cube core和24个vector core给主计算流，每条通信流分配12个vector core。注意力设备因干扰较大不启用三流设计。

#### 气泡免除核调度（MoE超级核）
- 异步执行导致MoE设备按非确定性顺序处理不同DP组的各层，传统CPU预发射核的方案会因为层ID动态解析而阻塞。MoE超级核将所有权重张量的地址列表预计算并存储到设备端，层ID作为动态设备端参数输入，核函数可直接索引目标权重，实现免等待的核发射，消除每层约220微秒的CPU调度开销。

### 2.3 算法流程（文字说明）
1. 请求到达后，调度器等待形成超过token数阈值的批对。
2. 批对被分配给空闲的注意力DP组。
3. 注意力设备对各层执行双批交错：计算层L的注意力 → 调用`async-dispatch-send` → 立即切换到下一个批次的层L注意力。
4. MoE设备轮询接收，对收到的层进行MoE计算（使用MoE超级核），然后调用`async-combine-send`返回结果。
5. 注意力设备累加结果后继续下一层，直到所有层完成，生成第一个token后转发给解码实例。

- **工作负载**：生产日志中序列长度分布（31~10万tokens，平均5k），请求到达服从泊松分布。仅保留≤32k的请求（超长请求用序列并行单独处理）。

## 6. 主要结论与发现
- ASAP在SLO合规吞吐量上比Default提升194%（2.94倍），比ChunkedPrefill提升90%（1.9倍）。
- 低负载下平均TTFT降低34.3%（350ms vs 533ms），高负载下降低68.3%。
- 异步通信原语延迟比同步P2P低4~5.8倍（根据token数）。
- 延迟分解显示：短请求在同步系统中受同步等待和排队延迟拖累严重（占55%+30%），ASAP将非核延迟降低最多80%。
- 消融实验证明双批交错提升14.3%吞吐，通信-计算重叠提升12.4%，MoE超级核提升约6%。

## 7. 优点
- **首次实现MoE预填充完全异步化**，彻底消除了全局同步屏障，是推理系统架构的重要创新。
- **高效的异步通信原语**：利用超节点共享内存缓冲区和标志位实现微秒级非阻塞传输，延迟远低于传统P2P。
- **系统级协同优化**：长度感知批处理、双批交错、三流重叠、气泡免除核调度四个机制相辅相成，有效克服了异步执行带来的碎片化和开销。
- **严格公平对比**：使用相同的设备总数（32 NPU）和同类模型参数，基线为最先进的同步系统。
- **实验分析深入**：包含延迟分解，清晰指出同步系统中短请求被“拖后腿”的量化证据。
- **开放讨论**：考虑了可移植性（RDMA）、解码阶段适用性（指出收益较小），表现出思考深度。
