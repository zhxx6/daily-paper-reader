## 摘要
我们介绍了Nemotron-Labs-3-Puzzle-75B-A9B，这是Nemotron-3-Super的压缩变体，专为交互式部署优化。我们设计该模型以在高用户吞吐约束下最大化服务器吞吐量。在单个8xB200节点的交互式服务工作负载中，Puzzle-75B-A9B在匹配的用户吞吐约束下，服务器吞吐量约为Nemotron-3-Super的2倍。在单个H100 GPU上的超长上下文部署中，压缩模型将1M token并发从1个请求提升至8个请求。Puzzle-75B-A9B通过多阶段流水线构建，该流水线结合了迭代Puzzle压缩框架、知识蒸馏、强化学习、量化和多标记预测头。压缩过程联合优化了异构MoE剪枝、激活参数预算和Mamba剪枝，以在保持模型质量的同时提高推理效率。我们在广泛的推理、编码、多语言、长上下文和智能体基准上评估了Puzzle-75B-A9B。尽管大幅压缩，该模型在众多任务中相对于父模型仍保持了强大的下游准确率。这些结果表明，大型混合MoE模型可以在保持强大下游能力的同时，针对部署效率进行大幅优化。我们的模型已在Hugging Face上公开发布。

## Abstract
We present Nemotron-Labs-3-Puzzle-75B-A9B, a compressed variant of Nemotron-3-Super optimized for interactive deployment. We designed the model to maximize server throughput under high user throughput constraints. In interactive serving workloads on a single 8xB200 node, Puzzle-75B-A9B achieves approximately 2x higher server throughput than Nemotron-3-Super at matched user throughput constraints. In ultra-long-context deployment on a single H100 GPU, the compressed model increases 1M-token concurrency from 1 request to 8 requests. Puzzle-75B-A9B is constructed using a multi-stage pipeline that combines the Iterative Puzzle compression framework with knowledge distillation, reinforcement learning, quantization, and a Multi-Token Prediction head. The compression process jointly optimizes heterogeneous MoE pruning, active parameter budget, and Mamba pruning to improve inference efficiency while preserving model quality. We evaluate Puzzle-75B-A9B on a broad suite of reasoning, coding, multilingual, long-context, and agentic benchmarks. Despite substantial compression, the model retains strong downstream accuracy relative to the parent model across a wide range of tasks. These results demonstrate that large hybrid MoE models can be substantially optimized for deployment efficiency while maintaining strong downstream capability. Our model is publicly available on Hugging Face.

## 1. 核心问题与整体含义（研究动机和背景）

*   **核心问题：** 大型混合专家（MoE）模型虽然性能强大，但面临严重的推理成本、高内存消耗和服务延迟问题，严重制约了其在生产环境下的交互式部署，尤其是在高用户吞吐和超长上下文（如百万token）的场景中。如何在不大幅降低模型质量的前提下，大幅提升推理效率，是当前部署下一代LLM的核心挑战。
*   **整体含义：** 本论文表明，通过系统性地应用硬件感知的压缩技术（包括剪枝、量化、知识蒸馏和投机解码），大型混合MoE模型可以被大幅优化部署效率，同时在广泛的任务中保持强大的下游能力。这为在实际应用中大规模部署前沿LLM提供了一条切实可行的路径。

## 2. 方法论：核心思想与关键技术细节

该方法论是一个 `结构压缩` + `性能恢复` 的**多阶段流水线**。

*   **核心思想：** 提出**迭代式Puzzle (Iterative Puzzle)** 框架，这是一种**顺序的、分层级的神经架构搜索**方法，用于在Deployment效率约束下选择最优的子架构。随后通过大规模知识蒸馏和强化学习“恢复”压缩带来的性能损失，最后辅以量化（FP8, NVFP4）和投机解码（MTP）技术进一步提升推理效率。

*   **核心技术细节：**
    *   **1. 结构压缩 (Iterative Puzzle):**
        *   **基础(Puzzle):** 给定一个预训练的“教师模型”，Puzzle定义了一个“拼图块”（可选层实现）的搜索空间，为每个块评估质量（精度/效率）得分，并用混合整数规划（MIP）选择最优的块组合，以满足目标约束。
        *   **迭代 (Iterative Puzzle):** 针对Puzzle的单步假设（即块替换的贡献是独立叠加的）可能忽略层间交互的不足，Iterative Puzzle采用了**渐进式压缩**：交替执行 “适度压缩（Puzzle搜索） -> 短期知识蒸馏（Healing）”。每次压缩后，当前模型成为下一次搜索的起点，使搜索能适应不断变化的内部表示。
        *   **算法流程（文字说明）：**
            1.  初始模型 \(M_0\) (教师模型)。
            2.  循环直到达到目标架构：
                a.  在 \(M_{r-1}\) 的搜索空间内进行Puzzle搜索（在 \(M_{r-1}\) 上下文下评估候选块）。
                b.  求解优化问题，得到压缩后的中间模型 \(\tilde{M}_r\)。
                c.  使用教师模型 \(T\) 和短阶段KD对 \(\tilde{M}_r\) 进行恢复 (Healing)，得到 \(M_r\)。
        *   **关键剪枝技术:**
            *   **MoE异构通道剪枝：** 在**不同层**将每个路由专家的中间维度**不均匀地**剪枝到不同大小，由Puzzle决定。
            *   **Top-k剪枝：** 允许每层的激活专家数（k）**不均匀地**变化（最多从22剪到4-18个）。
            *   **Mamba SSM状态剪枝：** 基于贡献度估计，将SSM状态大小从128均匀剪枝到96。
    *   **2. 性能恢复 (Recovery):**
        *   **知识蒸馏 (KD):** 在迭代Puzzle阶段和最终恢复阶段使用，学生模型被训练以匹配教师模型（原始Super）的logits。混合使用30%预训练数据和70%SFT数据。序列长度从32K逐步扩展到512K。
        *   **强化学习 (RL):** 针对压缩后**软件工程**能力下降最显著的问题，采用**无KL惩罚**的RL进行后训练，重点恢复该能力。分为两个阶段（单步工具使用和端到端沙箱环境）。最终采用**Checkpoint平均**策略提升稳定性和性能。
    *   **3. 量化与投机解码:**
        *   **量化:** 应用后训练量化产生针对 **Blackwell GPU的NVFP4 (W4A4)** 和针对 **Hopper GPU的FP8 (W8A8)** 两种部署检查点。对Mamba SSM状态，NVFP4采用FP16+随机舍入以平衡性能与精度。
        *   **投机解码 (MTP):** 继承自父模型，使用**共享头**的MTP，可递归生成长草稿。核心创新是改进了MTP头训练，通过让模型**适应自回归草稿生成**与训练阶段的教师强迫模式不匹配的情况，从而显著提高了中后段草稿的接受率（平均接受长度提升25-30%）。

## 7. 优点

*   **系统性与综合性：** 不是单一技术的使用，而是将硬件感知NAS（Iterative Puzzle）、多种剪枝、知识蒸馏、强化学习、量化、投机解码整合成一个**多阶段联合优化流水线**，思路清晰，工程实践度高。
*   **务实且深度部署导向：** 从头到尾都考虑了实际部署中的瓶颈（预填计算、解码IO、内存限制），并针对不同场景（B200交互式、H100长上下文）进行优化。
*   **创新的迭代思想：** **Iterative Puzzle**为解决NAS中“架构搜索”和“模型价值”不一致的问题提供了一个实用且有效的解决方案，通过“压缩-恢复”的迭代，让搜索过程能感知模型变化。

## 8. 不足与局限

*   **性能损失仍然存在：** 尽管保持了高精度，但在一些指令遵循、智能体任务上，与教师模型相比仍有明显差距（约2-3个百分点），表明压缩对更复杂、精细的任务仍有负面影响。
*   **RL恢复范围有限：** RL仅专注于恢复“软件工程”能力，但压缩可能导致其他领域（如创意写作、角色扮演）的微妙退化未被充分覆盖。论文也承认“RL的影响很小”，暗示了当前RL技术整体在压缩恢复上的局限性。
*   **评估缺少某些维度：** 没有评估多轮对话、立场检测、安全性和偏见等维度。这些在压缩模型中可能表现不同，存在潜在的部署风险。
*   **缺乏零样本/少样本小设置评估：** 论文所有基准主要是使用工具或长格式生成的，没有提供传统的零样本/少样本小设置（如HellaSwag, ARC等）的对比结果。这使得对模型基础能力的评估不完整。
*   **训练效率的缺失：** 虽然推理效率是核心，但压缩流水线本身需要**大量的计算资源**（如100B token的KD、大量的RL轨迹）。论文没有讨论训练成本，这可能对资源有限的组织构成障碍。
*   **架构依赖性：** 其剪枝和量化方案高度依赖于NVIDIA的特定GPU架构（B200, H100）以及其定制的PyTorch推理库（Puzzletron, Megatron-LM），方法的可迁移性有限。
